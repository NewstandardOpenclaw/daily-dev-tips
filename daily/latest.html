Laravel

サブトピックA: Eloquent/SQL最適化。N+1をwith()で潰し、重い一覧は必要列のみselect()で取得。
サブトピックB: サービス層分離。Controllerを薄くし、ビジネスロジックをService/Actionへ寄せる。
なぜ面白い？ 同じ機能でもSQL設計と責務分離で、体感速度と保守性が一気に上がる。
現場の使いどころ: 管理画面一覧、検索API、バッチ処理の肥大化対策。
ハマりどころ: eager loadしすぎでメモリ増、Service乱立で責務が再分散。
ミニ課題: 既存一覧APIのクエリ数をDebugbarで計測し、1つ改善PRを作成。
公式: Eloquent Relationships / Service Container

$users = User::query()->select(['id','name'])->with('roles:id,name')->latest()->paginate(20);

Laravelユニットテスト

対象は「純粋ロジック」をUnit、「HTTP/DB連携」をFeatureで分離すると速く安定。
なぜ面白い？ テストの意図が明確になり、壊れた箇所を数分で特定しやすい。
現場の使いどころ: 料金計算、ステータス遷移、権限判定など仕様の核。
ハマりどころ: モック過多で実装変更に弱い、Factory依存で実行が重い。
ミニ課題: 失敗しやすい1ロジックをデータプロバイダ化し境界値ケースを3件追加。
公式: Laravel Testing / Pest

it('rejects expired coupon', fn() => expect($service->canUse($coupon))->toBeFalse());

Vue（Vue2中心 + Vue3補足）

Vue2ではOptions APIの責務分離（computed優先、watch最小化）が保守の要。
Vue3補足: Composition APIでロジック共通化、ref/reactiveの使い分けを明示。
なぜ面白い？ 同じUIでも状態設計の粒度でバグ率が大きく変わる。
現場の使いどころ: 検索フォーム、一覧フィルタ、モーダル管理。
ハマりどころ: Vue2のthis依存、Vue3移行時のライフサイクル差分。
ミニ課題: Vue2コンポーネント1つでwatchをcomputedに置換し差分検証。
公式: Vue2 Guide / Vue3 Guide

Docker

開発はdocker compose watchやvolumeで反復速度を上げる。
なぜ面白い？ 環境差分トラブルをコード化して消せる。
現場の使いどころ: 新規メンバーオンボード、CI一致、依存固定。
ハマりどころ: キャッシュ不整合、UID/GID権限、巨大イメージ化。
ミニ課題: Dockerfileをマルチステージ化しサイズ差を記録。
公式: Docker Compose / Multi-stage build

AWS（開発者向け基礎→実務）

基礎: IAM最小権限、CloudWatchログ、S3/EC2/RDS役割理解。
実務: ALB+ECS/Fargate、Parameter Store、監視アラーム運用。
なぜ面白い？ 小さな設計判断が運用コストに直結する。
現場の使いどころ: API配備、自動スケール、障害一次対応。
ハマりどころ: 権限過多、タグ未整理、課金可視化不足。
ミニ課題: 1サービスにアラーム3種(5xx/Latency/CPU)追加。
公式: IAM / CloudWatch

React

再利用ロジックはCustom Hookへ、描画責務はコンポーネントへ分離。
なぜ面白い？ UI変更に強く、テストも小粒化できる。
現場の使いどころ: フォーム、無限スクロール、権限表示制御。
ハマりどころ: useEffect乱用、依存配列ミス、不要再レンダリング。
ミニ課題: 1画面で派生stateを削除しメモ化方針を見直す。
公式: React Learn / You Might Not Need an Effect

TypeScript

なぜ面白い？ 型安全を本番データにも拡張できる。

ハマりどころ: any混入、型アサーション乱用、過度なジェネリクス。
ミニ課題: 既存APIレスポンス1つをschema検証付きへ置換。

PHP関数

array_map/filter/reduceは意図が明確なら強力、複雑ならforeach優先。
なぜ面白い？ 関数選択だけでコードレビューの通過率が上がる。
現場の使いどころ: 変換パイプライン、集計、整形処理。
ハマりどころ: クロージャ深掘りで可読性低下、キー欠落バグ。
ミニ課題: 同処理をreduce版/foreach版で書いて比較。

配列ヘルパー

Laravelのdata_get/data_set/Arr::onlyで深い構造を安全に扱う。
なぜ面白い？ nullチェック地獄を避け、事故率を下げられる。
現場の使いどころ: APIレスポンス整形、設定配列抽出、DTO前処理。
ハマりどころ: ワイルドカード過信、デフォルト値の型崩れ。
ミニ課題: ネスト配列処理1箇所をhelper中心へ置換。

Lodash

ハマりどころ: deep clone乱用で性能劣化、tree-shaking不足。

HTML&CSS（必須基礎）

HTMLは意味論（header/main/nav/button/label）を守るとアクセシビリティが上がる。
CSSはレイアウトをFlex/Gridで整理し、余白はルール化。
なぜ面白い？ 基礎が強いほどJS実装が軽くなる。
現場の使いどころ: フォーム、レスポンシブ、デザイン崩れ修正。
ハマりどころ: div過多、詳細度競合、固定値乱用。
ミニ課題: 既存フォームにlabel/ariaとフォーカススタイル追加。

Zapier+n8n比較

Zapierは導入が速く、n8nは柔軟性・自己ホスト性が強み。
なぜ面白い？ 自動化の“速度”と“自由度”のトレードオフを体感できる。
現場の使いどころ: 通知連携、データ同期、承認フロー。
ハマりどころ: Zapierはタスク課金、n8nは運用責任が増える。
ミニ課題: 同一ワークフローを両方で試作し工数比較。
公式: Zapier Help / n8n Docs

OpenClaw

Cronは時刻厳守、Heartbeatはまとめ確認に向く。役割分担で通知疲れを減らす。
なぜ面白い？ 同じ自動化でも運用設計で満足度が変わる。
現場の使いどころ: 朝定期配信、障害チェック、日次レポート。
ハマりどころ: タスク粒度過多、重複通知、プロンプト肥大。
ミニ課題: 既存定期処理をCron/Heartbeatに再分類して1つ統合。
公式: OpenClaw Docs / OpenClaw GitHub
