境界づけられたコンテキスト（Bounded Context）
  • 定義: ドメインモデルが一貫した意味を保てる範囲を明示する設計境界。
  • 重要性: 同じ用語の意味衝突を防ぎ、チーム間の仕様齟齬を減らせる。
  • 使い分け: モノリス内でも『請求』『配送』のように文脈を分ける。DB分離は必須ではなく、まず言語境界を分ける。
  • よくある誤解: マイクロサービス化しないと意味がないという誤解。
  • 3分要点: 用語集を文脈ごとに作り、曖昧語を減らすだけでも効果が大きい。

参考: DDD Reference / Martin Fowler

サーキットブレーカー
  • 定義: 連続失敗時に外部呼び出しを一時遮断し、回復後に段階再開する保護機構。
  • 重要性: 下流障害の巻き込みを防ぎ、全体障害を回避できる。
  • 使い分け: リトライと併用しつつ、失敗率閾値でOpenにする。内部DB呼び出しより外部APIで優先適用。
  • よくある誤解: タイムアウト設定があれば不要という誤解。
  • 3分要点: Closed/Open/Half-Openの3状態遷移を図で確認する。

参考: Azure Pattern / resilience4j

悲観ロックと楽観ロック
  • 定義: 同時更新競合を、事前排他（悲観）か検出再試行（楽観）で制御する手法。
  • 重要性: 在庫・残高の不整合を防ぎつつ、性能劣化を避ける判断ができる。
  • 使い分け: 競合頻度が高いなら悲観、低いなら楽観が有利。
  • よくある誤解: 楽観ロックはロックしないので安全性が低いという誤解。
  • 3分要点: 更新失敗時の再試行UXまで含めて設計する。

参考: PostgreSQL Locking / JPA Optimistic Lock

Read Through / Write Through Cache
  • 定義: キャッシュを読み書き経路に組み込む一貫化パターン。
  • 重要性: DB負荷を下げつつ、キャッシュ不整合の事故を減らせる。
  • 使い分け: 更新直後の整合性が重要ならWrite Through、読み偏重ならRead Through中心。
  • よくある誤解: TTLを短くすれば整合性問題は解決するという誤解。
  • 3分要点: 失効戦略（キー粒度・イベント連動）を先に決める。

参考: AWS Caching Patterns / Redis

イベントソーシング入門
  • 定義: 現在状態ではなく、発生したイベント列を正として保存する設計。
  • 重要性: 監査可能性と状態再構築性が高まり、業務履歴を強く扱える。
  • 使い分け: 履歴価値が高い領域に限定導入し、CRUD領域へ無理に拡大しない。
  • よくある誤解: イベントを保存すればCQRSも必須という誤解。
  • 3分要点: まず1集約でリプレイ可能性を試す。

参考: Event Sourcing / CQRS Journey

分散トレーシングのスパン設計
  • 定義: リクエストを跨ぐ処理をSpanとして構造化し、遅延の発生点を追跡する手法。
  • 重要性: 『遅い』を感覚でなく区間単位で特定できる。
  • 使い分け: HTTP入口・DB・外部API・キュー投入の主要境界にSpanを置く。
  • よくある誤解: 全関数にSpanを入れるほど良いという誤解。
  • 3分要点: 高コスト区間だけにタグ付きSpanを入れて始める。

参考: OpenTelemetry / Jaeger

レートリミットのアルゴリズム比較
  • 定義: 固定窓・スライディング窓・トークンバケットで流量制御する実装選択。
  • 重要性: 過負荷防止と公平性を両立できる。
  • 使い分け: 瞬間スパイク吸収はトークンバケット、厳密公平はスライディング窓。
  • よくある誤解: 1ユーザー当たり上限だけ決めれば十分という誤解。
  • 3分要点: キー粒度（IP/ユーザー/APIキー）と返却ヘッダを設計する。

参考: Cloudflare Rate Limiting / NGINX Limit Req

機密情報のシークレットローテーション
  • 定義: 資格情報を定期更新し、古い秘密を失効させる運用。
  • 重要性: 漏洩時の被害期間を短縮できる。
  • 使い分け: 静的env埋め込みを避け、Vault/Secrets Manager経由で注入する。
  • よくある誤解: 年1回更新で十分という誤解。
  • 3分要点: 自動ローテーション + 破綻時ロールバック手順を用意。

参考: AWS Secrets Manager / HashiCorp Vault

品質ゲートとしての静的解析
  • 定義: マージ前に静的解析で潜在バグ・設計劣化を機械検出する運用。
  • 重要性: レビュー負荷を減らし、初期段階で欠陥を抑制できる。
  • 使い分け: 重大ルールのみfail、軽微は警告で段階導入。
  • よくある誤解: 警告ゼロを目指すほど品質が高いという誤解。
  • 3分要点: 最初は高インパクトルール3つに絞る。

参考: SonarQube / ESLint

リリース後検証（Post-deploy Verification）
  • 定義: デプロイ直後に主要機能と指標を短時間で確認する手順。
  • 重要性: 障害検知を早め、影響拡大前にロールバック判断できる。
  • 使い分け: SLI・エラーレート・主要導線APIをチェックリスト化して毎回実施。
  • よくある誤解: 監視アラートがあるので不要という誤解。
  • 3分要点: 5分で終わる最小検証セットを定義する。

参考: Google SRE Workbook / Deployment Strategies
