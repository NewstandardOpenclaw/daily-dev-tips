Laravel（今日のサブトピック: Eloquentの集計最適化 / SQLインデックス設計 / Queue再試行戦略）
  • 概要: Eloquentで読みやすく書きつつ、SQL側で速度を作り、重処理はQueueに逃がす三段構え。
  • なぜ面白いか: アプリコード・DB・非同期処理がつながって、体感速度が一気に変わる。
  • 現場での使いどころ: 管理画面の月次集計で withCount() + 複合index + バッチ化が効く。
  • ハマりどころ: Eloquentの便利メソッドだけで押し切ると、実行計画が悪化して遅くなる。
  • ミニ課題: 既存一覧API1本で「クエリ数」「p95応答」「EXPLAIN」を改善前後で比較。
  • 補足: 集計はselectRawで明示、Queueはtries/backoffを業務SLAに合わせる。
$users = User::query()
  ->withCount(['orders as paid_orders_count' => fn($q) => $q->where('status', 'paid')])
  ->orderByDesc('paid_orders_count')
  ->limit(20)
  ->get();

Laravelユニットテスト（境界値テストとテストデータ設計）
  • 概要: 正常系だけでなく境界値を押さえると、仕様漏れを早期に防げる。
  • なぜ面白いか: 小さなテスト追加で、将来の改修速度が目に見えて上がる。
  • 現場での使いどころ: 料金計算・在庫判定・有効期限判定など条件分岐の多い処理。
  • ハマりどころ: Factoryの初期値依存で、テストが暗黙仕様だらけになる。
  • ミニ課題: 既存Serviceに「最小値・最大値・不正値」の3ケースを追加してCI実行。
  • 補足: テスト名は「条件_期待結果」で揃えるとレビューが速い。

参考: Laravel Testing / PHPUnit Docs

Vue（Vue2主軸: computed責務分離 + Vue3 Composition API補足）
  • 概要: Vue2ではcomputedを中心に据え、watchは副作用だけに限定する。
  • なぜ面白いか: 依存関係が整理され、レンダリングの不具合原因を追いやすくなる。
  • 現場での使いどころ: 検索フォームで表示加工はcomputed、API再取得はwatchで分離。
  • ハマりどころ: deep watch多用でパフォーマンス低下と意図しない再実行が起きる。
  • ミニ課題: 既存watchを1つcomputedに置換し、再描画回数を比較。
  • Vue3補足: ref/computed/watch に置き換えても責務の分け方は同じ。

参考: Vue2 Computed & Watch / Vue3 Watchers

Docker（開発速度を落とさないイメージ設計）
  • 概要: 依存インストール層を先に固定し、アプリ層を後ろに置くと再ビルドが速い。
  • なぜ面白いか: 毎日の待ち時間が減るだけで、開発体験がかなり改善する。
  • 現場での使いどころ: PHP+Node混在のプロジェクトでチーム全員の環境差分を解消。
  • ハマりどころ: COPY . . を早く置きすぎてキャッシュが毎回無効化される。
  • ミニ課題: DockerfileのCOPY順を見直し、3回ビルド平均を記録。
  • 補足: 開発はvolume活用、本番は固定タグとマルチステージで軽量化。

参考: Docker Build Cache / Multi-stage builds

AWS（基礎から実務へ: IAM最小権限→CloudWatch運用）
  • 概要: 開発初期はIAMとネットワーク境界を固め、運用でCloudWatch監視を足す。
  • なぜ面白いか: セキュリティと運用の両面を、最小構成から段階的に積み上げられる。
  • 現場での使いどころ: APIをALB+ECSで運用し、エラー率・遅延をダッシュボード化。
  • ハマりどころ: 権限を広く取りすぎて監査時に是正コストが膨らむ。
  • ミニ課題: 既存ロール1つから不要アクションを削除し、CloudWatchアラームを1つ追加。
  • 補足: まずは「5xx率」「CPU」「メモリ」「再起動回数」を共通指標にする。

参考: IAM User Guide / CloudWatch

React（状態の持ち場所を設計する）
  • 概要: 「どこにstateを持つか」を先に決めると、props設計が一気に楽になる。
  • なぜ面白いか: コンポーネント分割の質が上がり、再利用性が自然に高まる。
  • 現場での使いどころ: フィルター付き一覧で、親に検索条件・子に表示責務を分離。
  • ハマりどころ: 兄弟間で同じstateを持って同期ズレを起こす。
  • ミニ課題: 1画面でstateを親へ集約し、子を可能な限り純粋表示に変更。
  • 補足: 必要になるまでContextを増やしすぎないのが安定運用のコツ。

TypeScript（実装前に型で境界を決める）
  • 概要: API入出力型を先に定義すると、仕様ブレをコンパイル時に検知できる。
  • なぜ面白いか: バグ修正より前に、バグが入る入口を狭められる。

  • ミニ課題: 外部API1本をunknown受け→型ガードへ置換して安全化。
  • 補足: 複雑なUnionは識別子フィールド追加で可読性が上がる。

PHP関数 / 配列ヘルパー（array系とCollectionの使い分け）
  • 概要: 単純加工はarray関数、複雑パイプラインはCollectionで役割分担する。
  • なぜ面白いか: 処理意図が読みやすくなり、レビュー時の認知負荷が下がる。
  • 現場での使いどころ: CSV整形・APIレスポンス整形・画面表示用のグルーピング。
  • ハマりどころ: ネストしたarray_mapの連鎖で可読性が急落する。
  • ミニ課題: 既存の配列加工1箇所をCollection化し、行数と可読性を比較。
  • 補足: パフォーマンスが重要な箇所はプロファイルを取って判断する。
$total = collect($orders)
  ->where('status', 'paid')
  ->sum('amount');

Lodash（必要最小限importで実用する）
  • 概要: debounce/throttle/groupByなど効果の高い関数だけを限定利用する。
  • なぜ面白いか: 体感UXを改善しつつ、バンドル肥大化も抑えられる。
  • 現場での使いどころ: 検索入力やスクロール監視でイベント連打を制御。
  • ハマりどころ: import _ from 'lodash' で不要コードまで同梱される。
  • ミニ課題: 全量importを関数単位importに置換し、bundle差分を確認。
  • 補足: ネイティブ代替できる箇所は積極的に置換して依存を軽くする。

参考: Lodash Docs / Tree Shaking

OpenClaw（毎日運用で効く設計）
  • 概要: cronで定時実行、sub-agentで重い作業分離、mainで判断を行う設計が安定。
  • なぜ面白いか: 自動化しつつ、人間判断のポイントを残せる。
  • 現場での使いどころ: 日次レポート配信、障害調査、複数タスクの並列処理。
  • ハマりどころ: 通知を増やしすぎて重要アラートが埋もれる。
  • ミニ課題: 1つの定型タスクをcron化し、通知文を80文字以内に最適化。
  • 補足: まずは「重要のみ即通知・他はまとめ報告」で運用開始が無難。
