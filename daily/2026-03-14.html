Idempotency Key設計
  • これは何か: 同じリクエストの重複実行を防ぐために、クライアントが一意キーを添えて結果を再利用する設計。
  • なぜ重要か: 決済や注文で二重処理を防ぎ、障害時の再送を安全にできる。
  • 使いどころ/使い分け: 書き込みAPIで必須。読み取りAPIでは通常不要。短期保持キーか永続キーかを業務要件で選ぶ。
  • よくある誤解: DBのUNIQUE制約だけで十分だと思うこと。レスポンス再返却戦略まで必要。
  • 3分要点: キーの寿命・スコープ・再送時レスポンス方針の3点を先に決める。

参考: Martin Fowler / AWS Builders' Library

Feature Flag運用
  • これは何か: 機能の有効/無効をデプロイとは独立して切り替える仕組み。
  • なぜ重要か: 段階リリースや緊急停止ができ、失敗時の影響を最小化できる。
  • 使いどころ/使い分け: 実験機能や高リスク変更で使う。恒久条件分岐には使わず、期限付きで運用する。
  • よくある誤解: フラグを増やすほど安全になるという誤解。放置フラグは複雑性を増やす。
  • 3分要点: 作成時に削除日を決め、オーナーを明確化する。

参考: Martin Fowler / AWS Builders' Library

バックプレッシャー
  • これは何か: 下流の処理能力に合わせて上流の流量を抑える制御。
  • なぜ重要か: キュー詰まりやタイムアウト連鎖を防ぎ、全体の安定性を守る。
  • 使いどころ/使い分け: ストリーム処理や非同期パイプラインで有効。スロットリングは利用者単位、バックプレッシャーは系全体制御。
  • よくある誤解: キューを大きくすれば解決するという誤解。遅延を隠すだけで根本対策にならない。
  • 3分要点: 拒否・遅延・縮退の3モードを用意して設計する。

参考: Martin Fowler / AWS Builders' Library

契約テスト（Contract Test）
  • これは何か: サービス間のAPI契約を自動検証し、互換性を守るテスト。
  • なぜ重要か: 統合環境でしか気づけない破壊変更を早期検出できる。
  • 使いどころ/使い分け: マイクロサービス間連携で使う。E2Eの代替ではなく、境界保証を担う。
  • よくある誤解: スキーマ一致だけ見れば十分という誤解。意味論的互換性も確認が必要。
  • 3分要点: Consumer主導で期待を明文化し、Provider側CIで検証する。

参考: Martin Fowler / AWS Builders' Library

SLOとError Budget
  • これは何か: 目標可用性（SLO）と許容失敗量（Error Budget）で運用判断する考え方。
  • なぜ重要か: 感覚ではなく数値でリリース速度と安定性のバランスを取れる。
  • 使いどころ/使い分け: 高信頼サービスで有効。SLAは対外契約、SLOは内製運用目標として使い分ける。
  • よくある誤解: 99.9%を目標にすれば安心という誤解。ユーザー影響の大きい指標を選ぶ必要がある。
  • 3分要点: まず重要ユーザージャーニーを1つ定義し、SLI→SLOを設定する。

参考: Martin Fowler / AWS Builders' Library

データマイグレーションのExpand/Contract
  • これは何か: 互換期間を設けて段階的にスキーマ移行する手法。
  • なぜ重要か: 本番停止や同時切替失敗を避け、安全に構造変更できる。
  • 使いどころ/使い分け: カラム改名や型変更で有効。先に追加（Expand）し、移行後に削除（Contract）する。
  • よくある誤解: 一回のDDLで終わらせるほうが速いという誤解。復旧不能リスクが高い。
  • 3分要点: アプリ互換期間を含めて2〜3リリースで計画する。

参考: Martin Fowler / AWS Builders' Library

N+1問題の検知
  • これは何か: 親データ取得後に子データを都度問い合わせてしまう性能劣化問題の検知。
  • なぜ重要か: データ量増加時に突然遅くなる原因を事前に潰せる。
  • 使いどころ/使い分け: ORM利用時に特に重要。JOIN最適化・バッチ取得・キャッシュを使い分ける。
  • よくある誤解: ローカルで速いから問題ないという誤解。本番データ量で顕在化する。
  • 3分要点: クエリ件数メトリクスをテストと監視に組み込む。

参考: Martin Fowler / AWS Builders' Library

Outboxパターン
  • これは何か: DB更新とメッセージ発行を同一トランザクション境界で整合させる実装パターン。
  • なぜ重要か: 『DBは更新されたがイベント未送信』を防ぎ、非同期連携を信頼できる。
  • 使いどころ/使い分け: イベント駆動連携で使う。2PCより実装しやすく現実的。
  • よくある誤解: Outboxテーブルを作れば終わりという誤解。再送・重複排除設計が必要。
  • 3分要点: 送信ワーカーの監視と再試行上限を必ず設計に含める。

参考: Martin Fowler / AWS Builders' Library

指数バックオフとジッター
  • これは何か: 再試行間隔を指数的に伸ばしつつ、ランダム揺らぎを加える失敗耐性手法。
  • なぜ重要か: 障害時の同時再試行集中（再試行ストーム）を防げる。
  • 使いどころ/使い分け: 外部API呼び出しで使う。固定間隔リトライよりも混雑回避に強い。
  • よくある誤解: 回数だけ決めればよいという誤解。最大待機時間と打ち切り条件も必要。
  • 3分要点: retry条件・上限・ジッター方式をセットで定義する。

参考: Martin Fowler / AWS Builders' Library

依存関係の健全性監査
  • これは何か: ライブラリの脆弱性・更新停滞・ライセンスを定期点検する運用。
  • なぜ重要か: 技術的負債を可視化し、事故前に改善の優先順位を付けられる。
  • 使いどころ/使い分け: 月次やリリース前に実施。重大脆弱性対応と通常更新をレーン分離する。
  • よくある誤解: 最新版追従が常に正義という誤解。互換性検証コストとのバランスが必要。
  • 3分要点: 『更新ポリシー』を文章化してチームで共有する。

参考: Martin Fowler / AWS Builders' Library

質問テンプレ: 「依存関係の健全性監査について、定義・重要性・使い分け・注意点を初心者にもわかるように整理して。」
