CRDTの基本
  • これは何か: 競合しても最終的に整合するよう設計された分散データ構造。
  • なぜ重要か: オフライン編集やマルチリージョン更新で衝突解決を自動化できる。
  • 使いどころ/使い分け: 共同編集や同期アプリで有効。厳密順序が必要な会計処理には不向き。
  • よくある誤解: マージできるなら何でも正しいという誤解。業務上の意味が壊れる場合がある。
  • 3分要点: 可換性・冪等性・結合性の3性質を理解する。

参考: Martin Fowler / AWS Builders' Library

Hedged Request
  • これは何か: 一定時間応答がないリクエストを別経路で追加送信し、遅延尾部を短縮する手法。
  • なぜ重要か: 平均値では見えないp99遅延を改善し、体感速度を上げられる。
  • 使いどころ/使い分け: 読み取り系で有効。書き込み系は副作用重複に注意して原則避ける。
  • よくある誤解: 常に二重送信すれば速いという誤解。負荷悪化を招く。
  • 3分要点: 遅延閾値を設けて条件付きで発火させる。

参考: Martin Fowler / AWS Builders' Library

WAL（Write-Ahead Logging）
  • これは何か: 本体反映前に変更ログを先に永続化する仕組み。
  • なぜ重要か: 障害時の復旧性を高め、整合性を守れる。
  • 使いどころ/使い分け: DB内部だけでなく独自ストレージ実装でも有効。
  • よくある誤解: ログがあるならバックアップ不要という誤解。用途が異なる。
  • 3分要点: 復旧手順（redo/undo）までセットで設計する。

参考: Martin Fowler / AWS Builders' Library

Capability-based Security
  • これは何か: 権限を『誰か』ではなく『何ができるトークンか』で表すセキュリティモデル。
  • なぜ重要か: 最小権限を実装しやすく、権限委譲を安全に扱える。
  • 使いどころ/使い分け: マイクロサービス間の限定権限委譲で有効。RBACと併用する。
  • よくある誤解: トークンを発行すれば安全という誤解。失効とスコープ管理が核心。
  • 3分要点: 短寿命・限定スコープ・監査ログの3点を徹底する。

参考: Martin Fowler / AWS Builders' Library

Tail Latency分析
  • これは何か: 平均ではなく高パーセンタイル遅延を重視して性能を評価する考え方。
  • なぜ重要か: 一部の遅いリクエストが全体満足度を大きく下げるため。
  • 使いどころ/使い分け: APIや検索で重要。p50改善よりp99改善を優先する場面が多い。
  • よくある誤解: 平均応答が速い=ユーザー体感も良いという誤解。
  • 3分要点: ボトルネックを『頻度×遅さ』で評価する。

参考: Martin Fowler / AWS Builders' Library

セルベースアーキテクチャ
  • これは何か: システムを小さな独立セルに分け、障害影響を局所化する設計。
  • なぜ重要か: 大規模障害の爆発半径を抑え、運用回復を早める。
  • 使いどころ/使い分け: マルチテナント基盤で有効。共有基盤との境界設計が鍵。
  • よくある誤解: 単にクラスタを増やせば同じという誤解。運用独立性が重要。
  • 3分要点: データ・制御・監視の分離単位を先に決める。

参考: Martin Fowler / AWS Builders' Library

Snapshot Testingの限界
  • これは何か: UI出力全体比較で回帰検知するテスト手法と、その運用課題。
  • なぜ重要か: 差分が多すぎるとレビュー疲れで品質低下を招くため。
  • 使いどころ/使い分け: コンポーネントの静的表示で有効。動的値はマスクして使う。
  • よくある誤解: スナップショットが通れば安心という誤解。意図の検証は別途必要。
  • 3分要点: 重要部分を絞ったアサーションに併用する。

参考: Martin Fowler / AWS Builders' Library

Work Stealing
  • これは何か: 空いているワーカーが他キューの仕事を奪って処理する並列実行方式。
  • なぜ重要か: 偏りを減らし、CPU利用率を高く維持できる。
  • 使いどころ/使い分け: 不均一タスクで有効。リアルタイム厳格順序には注意。
  • よくある誤解: 常に公平になるという誤解。大粒度タスクでは偏りが残る。
  • 3分要点: タスク粒度を小さく設計すると効果が出やすい。

参考: Martin Fowler / AWS Builders' Library

Schema Registry運用
  • これは何か: イベント/メッセージのスキーマを集中管理し互換性を統制する仕組み。
  • なぜ重要か: ProducerとConsumerの独立デプロイを安全に進められる。
  • 使いどころ/使い分け: Kafka等のイベント基盤で有効。互換モードの選択が運用品質を左右する。
  • よくある誤解: 登録すれば破壊変更が起きないという誤解。レビュー規律が必要。
  • 3分要点: 後方互換を基本にし、例外は明示承認にする。

参考: Martin Fowler / AWS Builders' Library

データ指向設計（Data-Oriented Design）
  • これは何か: オブジェクトよりデータ配置とアクセスパターンを優先して性能を最適化する設計。
  • なぜ重要か: CPUキャッシュ効率を改善し、高頻度処理を高速化できる。
  • 使いどころ/使い分け: ゲームや数値処理で有効。業務アプリ全体に無理適用しない。
  • よくある誤解: 低レベル最適化だけの話という誤解。設計段階のデータ分割が本質。
  • 3分要点: 『どの順でデータを読むか』から構造を決める。

参考: Martin Fowler / AWS Builders' Library
