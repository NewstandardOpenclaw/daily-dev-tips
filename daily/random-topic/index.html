Observability Lite（1時間で監視導線を作る）
  • 概要: メトリクス・ログ・アラートを最小構成で接続する。
  • なぜ面白いか: 障害時の「何見ればいい？」が消える。
  • 現場での使いどころ: 新規サービス立ち上げ初日。
  • ハマりどころ: 指標だけ作って通知先を決めない。
  • ミニ課題: 5xxアラートをSlackに飛ばす。
質問テンプレ:「最小の監視導線を1時間で組む手順を作って」
API契約テスト（壊さないAPI運用）
  • 概要: OpenAPIを基準にレスポンス互換性を検証する。
  • なぜ面白いか: フロント破壊をリリース前に止められる。
  • 現場: 複数クライアントがぶら下がるAPI。
  • ハマりどころ: サンプルJSONだけで安心する。
  • ミニ課題: 主要エンドポイント2本を契約テスト化。
質問テンプレ:「API契約テストの導入ステップを最短で教えて」
キャッシュ戦略（TTL設計の勘所）
  • 概要: 更新頻度と許容遅延からTTLを決める。
  • なぜ面白いか: DB負荷を落としつつ鮮度を維持できる。
  • 現場: 一覧API・ランキング表示。
  • ハマりどころ: 一律TTLで事故る。
  • ミニ課題: 1APIだけTTL最適化して計測。
質問テンプレ:「このAPIのTTLをどう決めるべきかレビューして」
Feature Flag運用（安全リリース）
  • 概要: 機能ON/OFFを環境・ユーザー単位で制御する。
  • なぜ面白いか: ロールバックが秒でできる。
  • 現場: 新機能の段階公開。
  • ハマりどころ: 使い終わったflagを放置。
  • ミニ課題: 期限付きflag運用ルールを作る。
質問テンプレ:「Feature Flagの運用ルールを簡潔に設計して」
DBマイグレーション（無停止変更）
  • 概要: expand/contractで段階的にスキーマ変更する。
  • なぜ面白いか: 本番停止リスクを減らせる。
  • 現場: カラム分割・型変更。
  • ハマりどころ: 一発変更でロック地獄。
  • ミニ課題: 既存変更案を2段階化してみる。
質問テンプレ:「このDB変更を無停止でやる手順に直して」
インシデント初動（15分テンプレ）
  • 概要: 影響範囲確認→暫定対応→連絡の順を固定する。
  • なぜ面白いか: 焦っても品質が崩れにくい。
  • 現場: 夜間障害・急な遅延。
  • ハマりどころ: 原因調査を先にやりすぎる。
  • ミニ課題: 初動テンプレをチームに共有。
質問テンプレ:「障害初動テンプレを15分運用向けに作って」
コードレビュー高速化（観点固定）
  • 概要: 可読性・安全性・性能の観点を固定する。
  • なぜ面白いか: レビュー品質が人依存しにくい。
  • 現場: PRが滞留している時。
  • ハマりどころ: 好みレビューになる。
  • ミニ課題: 5項目チェックリスト作成。
質問テンプレ:「レビュー観点を5項目に絞って作って」
Webセキュリティヘッダ（最低限セット）
  • 概要: CSP/XFO/XCTO/Referrer-Policyを整理導入。
  • なぜ面白いか: 小コストで事故リスクを下げる。
  • 現場: 公開Webアプリ全般。
  • ハマりどころ: CSPを厳しくしすぎて壊す。
  • ミニ課題: report-onlyでCSP試験運用。
質問テンプレ:「最小のセキュリティヘッダ構成を提案して」
クラウドコスト可視化（まず見る3指標）
  • 概要: 日次コスト・サービス別・急増アラートを先に置く。
  • なぜ面白いか: 「気づいたら高い」を防げる。
  • 現場: 小規模チームの運用。
  • ハマりどころ: 月次だけ見て遅れる。
  • ミニ課題: 前日比アラートを設定。
質問テンプレ:「小規模向けクラウドコスト監視を設計して」
AIワークフロー設計（人間の確認点を残す）
  • 概要: 自動化の前に人間レビューゲートを定義する。
  • なぜ面白いか: 速度と安全性を両立できる。
  • 現場: PR要約・テスト生成・一次分類。
  • ハマりどころ: 全自動化して誤判定が拡散。
  • ミニ課題: 承認必須の境界を明文化。
質問テンプレ:「AIワークフローに人間確認点を設計して」
