Raft合意アルゴリズムの勘所
  • 定義: 分散ノード間でログ順序を一致させるリーダー中心の合意手法。
  • 重要性: 分散環境でも一貫した状態更新を実現できる。
  • 使い分け: 設定ストアやメタデータ管理に向く。高スループット書き込みには構成検討が必要。
  • よくある誤解: 分散システムなら必ずRaftを使うべきという誤解。
  • 3分要点: Leader election・log replication・commit条件の3点を押さえる。

参考: Raft Paper / etcd docs

NUMAを意識した性能最適化
  • 定義: メモリアクセスの物理配置差を考慮してスレッドとメモリを配置する最適化。
  • 重要性: 高負荷サーバーでレイテンシぶれを減らせる。
  • 使い分け: DBや推論基盤などCPU密集ワークロードで有効。
  • よくある誤解: CPUコア数だけ増やせば速くなるという誤解。
  • 3分要点: まずnumactlとperfで現状観測する。

参考: Linux NUMA / numactl

LSM-TreeとB-Treeの選択
  • 定義: 書き込み最適のLSMと読み取り安定のB-Treeを特性で選ぶ考え方。
  • 重要性: ストレージ選択が性能・運用コストを大きく左右する。
  • 使い分け: 書き込み多量+圧縮重視ならLSM、ポイント参照重視ならB-Tree寄り。
  • よくある誤解: 新しい実装ほど常に優れているという誤解。
  • 3分要点: ワークロード比率（read/write/scan）を先に数値化する。

参考: RocksDB / InnoDB B-Tree

ゼロトラストの実装単位
  • 定義: ネットワーク境界を信用せず、通信ごとに認証・認可を行う設計思想。
  • 重要性: 内部侵害を前提に被害拡大を抑止できる。
  • 使い分け: IDベースアクセス制御、短命証明書、端末状態評価を組み合わせる。
  • よくある誤解: VPNがあればゼロトラスト達成という誤解。
  • 3分要点: 『誰が・どの端末で・何へ』を毎回検証する。

参考: NIST Zero Trust / Google BeyondCorp

eBPF観測の基礎
  • 定義: カーネル空間に安全なプログラムを注入して低オーバーヘッド観測する技術。
  • 重要性: 本番への影響を抑えながら深い可観測性を得られる。
  • 使い分け: ネットワーク遅延解析やシステムコール観測で有効。
  • よくある誤解: eBPFはトレース専用で実運用制御には使えないという誤解。
  • 3分要点: bcc/bpftraceで短いワンライナー観測から始める。

参考: eBPF Docs / bpftrace

PaxosとRaftの違いを実務で捉える
  • 定義: どちらも合意形成手法だが、理解容易性と実装エコシステムに差がある。
  • 重要性: 技術選定時の学習コストと運用性判断に直結する。
  • 使い分け: 既製実装が豊富なRaft採用が一般的。Paxosは理論理解の基礎として価値が高い。
  • よくある誤解: Paxosは古いので無価値という誤解。
  • 3分要点: 論文の難易度ではなく、運用可能な実装可否で選ぶ。

参考: Paxos Made Simple / Raft FAQ

メルクル木（Merkle Tree）活用
  • 定義: ハッシュ木でデータ集合の一致検証を効率化する構造。
  • 重要性: 大規模同期や改ざん検知の通信量を大幅に削減できる。
  • 使い分け: レプリケーション差分同期、ブロックチェーン検証に有効。
  • よくある誤解: ブロックチェーン専用技術という誤解。
  • 3分要点: 葉ハッシュ→親ハッシュ集約の階層構造を図で理解する。

参考: Cassandra Anti-Entropy / Bitcoin Merkle Trees

Actorモデルの設計原則
  • 定義: 状態を持つアクター間メッセージで並行処理を構成するモデル。
  • 重要性: 共有メモリ競合を避け、スケールしやすい実装にできる。
  • 使い分け: 非同期イベント処理や分散ワーカー設計に向く。
  • よくある誤解: Actorを使えばデッドロックが絶対起きないという誤解。
  • 3分要点: メッセージ契約と監督戦略（supervision）を明示する。

参考: Akka / Erlang OTP

カオスエンジニアリングの始め方
  • 定義: 意図的に障害を注入して、回復性の弱点を事前に発見する実験手法。
  • 重要性: 本番障害の『初見対応』を減らし、運用品質を上げられる。
  • 使い分け: 小さなblast radiusで仮説検証し、段階的に対象を広げる。
  • よくある誤解: 本番を壊す危険な遊びという誤解。
  • 3分要点: 仮説・停止条件・ロールバックを先に定義する。

参考: Principles of Chaos / Gremlin Learning

カーネルバイパス（DPDK）概念
  • 定義: カーネルネットワークスタックを迂回しユーザー空間で高速パケット処理する技術。
  • 重要性: 超低遅延・高スループットのネットワーク処理で効果が大きい。
  • 使い分け: トレーディング・NFV・高性能ゲートウェイで採用。汎用Webには過剰。
  • よくある誤解: どのサービスでも速くなる万能策という誤解。
  • 3分要点: 性能向上と開発運用複雑性のトレードオフを理解する。

参考: DPDK / Linux Networking
