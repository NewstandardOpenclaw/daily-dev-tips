Laravel（今日のサブトピック: Eloquent集計SQL / キャッシュ設計 / Queue再試行）
  • 概要: Eloquentで集計クエリを安全に書きつつ、結果をキャッシュし、重い処理はQueueで非同期化する3点セットを押さえる日。
  • なぜ面白いか: SQL最適化だけでなく「体感速度」と「障害耐性」を同時に上げられる。
  • 現場での使いどころ: ダッシュボードAPIの「遅い/重い/たまに落ちる」を段階的に改善できる。
  • ハマりどころ: Eloquentの便利さに頼りすぎて、実行SQLとindex設計を確認しないまま本番投入しがち。
  • 実務メモ: まずtoSql()とEXPLAINでボトルネック確認→Cache::remember→重処理をJobへ分離。
  • 今日の実践ミニ課題: 既存の集計APIを1本選び、クエリ時間・レスポンス時間・ジョブ失敗率をbefore/afterで記録する。

Laravelユニットテスト（Day 2: Mockeryと依存注入テスト）
  • 概要: 外部依存（API/メール/時刻）をモック化して、ビジネスロジックの純粋な振る舞いを検証する。
  • なぜ面白いか: テストが速く安定し、仕様変更への追従が圧倒的に楽になる。
  • 現場ユースケース: 決済成功時のみ通知ジョブを発行する分岐の安全確認。
  • ハマりどころ: モックしすぎて実装詳細テストになり、リファクタ時に壊れやすくなる。
  • 実務メモ: 「入出力契約」を検証し、内部呼び出し順序への過剰アサーションは避ける。
  • 今日の実践ミニ課題: サービス1件をconstructor DI化し、外部依存をFake/Mockに置き換えてテスト追加。

参考: Laravel Testing / Mockery

Vue（Vue2主軸: options APIの責務分離 + Vue3補足）
  • 概要: Vue2ではdata/methods/computed/watchの責務を分けると保守性が跳ね上がる。
  • なぜ面白いか: 「どこを読めば何が起きるか」が明確になり、デバッグ時間を短縮できる。
  • 現場での使いどころ: フォーム画面で入力整形はcomputed、保存トリガーはmethods、外部同期はwatchに限定。
  • ハマりどころ: watchで状態を書き換え続け、循環更新や意図しない再描画を起こす。
  • Vue3補足: Composition APIでも原則は同じで、computedは派生値、watchは副作用に絞る。
  • 今日の実践ミニ課題: 既存Vue2コンポーネント1つでwatchを1個削減し、computedへ移した差分を確認。

参考: Vue2 Components / Vue2 Computed & Watch / Vue3 Watchers

Docker（マルチステージビルドで軽量化）
  • 概要: build環境とruntime環境を分離するマルチステージで、イメージサイズと脆弱性面積を削減する。
  • なぜ面白いか: push/pull時間が短くなり、CI/CD全体の待ち時間が目に見えて減る。
  • 現場での使いどころ: Nodeビルド成果物だけをnginxにコピーするSPA配信構成。
  • ハマりどころ: build stageで生成したファイルパスをruntime stageに正しくコピーできず起動失敗。
  • 実務メモ: ベースイメージのタグ固定と.dockerignore整理はセットで実施。
  • 今日の実践ミニ課題: 既存Dockerfileをマルチステージ化し、イメージサイズ差分を記録。

参考: Multi-stage builds / Build cache

AWS（Day 2: S3 + CloudFront + IAM実務基礎）
  • 概要: 静的配信の定番であるS3 + CloudFrontを、最小権限IAMとあわせて実務目線で押さえる。

  • 現場での使いどころ: フロントエンド成果物の配信とAPI（ALB/Lambda）を分離する構成。
  • ハマりどころ: S3をpublic化しすぎる、CloudFront無効化忘れで古いJSを配信し続ける。
  • 実務メモ: OAC/OAIでS3直アクセスを塞ぎ、キャッシュ戦略はimmutableとセットで設計。
  • 今日の実践ミニ課題: 検証用バケット1つでprivate配信 + CloudFront経由アクセスを構築して確認。

Lodash（debounce/throttleの使い分け）
  • 概要: 入力イベントやスクロールイベントでdebounceとthrottleを目的別に使う。

  • 実務メモ: Lodash全量importではなく関数単位importでバンドル肥大化を抑制。
  • 今日の実践ミニ課題: 1画面でdebounce導入前後のAPIリクエスト回数を比較。

OpenClaw（cron + sub-agent運用の基本設計）
  • 概要: 定期実行はcron、重い処理はsub-agent、最終報告は短文通知に分離する設計が安定しやすい。
  • なぜ面白いか: 自動化を増やしても通知ノイズを抑え、運用負荷を制御できる。
  • 現場での使いどころ: 毎朝のレポート生成、依存サービスのヘルスチェック、定期バックアップ監視。
  • ハマりどころ: すべてをメインセッションで回して履歴が肥大化し、追跡しづらくなる。
  • 実務メモ: 失敗時のみ詳細ログ、成功時は要約1行にすると通知品質が上がる。
  • 今日の実践ミニ課題: 1つの定期ジョブを「実行処理」と「通知文生成」に分離して運用する。
