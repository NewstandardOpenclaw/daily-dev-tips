Laravel（今日のサブトピック: Eloquent集計SQL / FormRequest / トランザクション）
  • 概要: Eloquentで集計を組みつつ、入力検証はFormRequest、更新系はトランザクションで安全性を固める日。
  • なぜ面白いか: 「読めるコード」と「落ちにくい本番運用」の両方を同時に作れる。
  • 現場での使いどころ: 売上ダッシュボードAPIや一括更新処理で、整合性と速度を両立したい時に効く。
  • ハマりどころ: 集計列にインデックスが無いままgroupByだけ書いて遅延、検証漏れで不正入力が混入しがち。
  • ミニ課題: 既存API1本で「before/afterのSQL」「EXPLAIN」「失敗時ロールバック」を確認して差分を記録する。
  • 公式リンク: Eloquent / Query Builder / Validation / Database Transactions を合わせて読む。
$report = Order::query()
  ->selectRaw('user_id, COUNT(*) as cnt, SUM(total) as sum_total')
  ->whereBetween('created_at', [$from, $to])
  ->groupBy('user_id')
  ->orderByDesc('sum_total')
  ->limit(20)
  ->get();

参考: Laravel Eloquent / Query Builder / Validation / Transactions

Laravelユニットテスト（Mockとデータプロバイダ）
  • 概要: 依存サービスはMockで隔離し、データプロバイダで境界値をまとめて検証する。
  • なぜ面白いか: テストが短くてもカバレッジを広げやすく、仕様変更に強くなる。
  • 現場での使いどころ: 料金計算、権限判定、ステータス遷移のような分岐ロジックに最適。
  • ハマりどころ: Mockしすぎると実装詳細に依存して、リファクタ時に壊れやすい。
  • ミニ課題: 1つのサービステストをデータプロバイダ化して、case追加コストを体感する。
  • 公式リンク: Laravel Testing と PHPUnit Data Providers。

参考: Laravel Testing / PHPUnit Data Providers

Vue（Vue2主軸: ライフサイクル整理 + Vue3補足）
  • 概要: Vue2のcreated/mounted/beforeDestroyの責務を分け、Vue3のonMounted/onUnmountedへ橋渡しする。
  • なぜ面白いか: メモリリークや二重イベント登録を予防でき、保守時の事故が減る。
  • 現場での使いどころ: WebSocket購読、スクロール監視、外部ライブラリ初期化の管理で効果大。
  • ハマりどころ: mountedでaddEventListenerして、破棄時にremoveを忘れて挙動が重くなる。
  • ミニ課題: 既存コンポーネント1つでイベントの登録/解除を見直し、Vue3書き換え案も併記する。
  • 公式リンク: Vue2 Lifecycle と Vue3 Lifecycle Hooks。

参考: Vue2 Lifecycle / Vue3 Lifecycle Hooks

Docker（マルチステージビルドで軽量化）
  • 概要: build用とruntime用イメージを分離して、本番イメージを小さく保つ。
  • なぜ面白いか: pull/push時間、脆弱性スキャン時間、起動時間をまとめて短縮できる。
  • 現場での使いどころ: Nodeビルド成果物だけをNginxに載せる、PHP拡張のビルド成果だけ移送する場面。
  • ハマりどころ: キャッシュを意識せずCOPY順が崩れ、毎回フルビルドになる。
  • ミニ課題: 既存Dockerfileを2段階化し、サイズ差とbuild時間差を記録する。
  • 公式リンク: Docker Multi-stage builds / Build cache。

参考: Docker Multi-stage / Docker Build Cache

AWS（開発者向け: CloudWatchで運用の見える化）
  • 概要: メトリクス/ログ/アラームを最小構成で作り、障害検知と原因調査を速くする。
  • なぜ面白いか: インフラ知識が薄くても「壊れた時に追える環境」を作れるようになる。
  • 現場での使いどころ: ALB 5xx増加、EC2 CPU高騰、Lambdaエラー急増をチームに即通知する。
  • ハマりどころ: アラート閾値が厳しすぎて通知疲れ、逆に緩すぎて見逃しが起きる。
  • ミニ課題: 1サービス分の主要メトリクス3つを選び、CloudWatch Alarmを作ってSNS通知を接続する。
  • 公式リンク: CloudWatch User Guide と Well-Architected。

参考: Amazon CloudWatch / AWS Well-Architected

React（状態の持ち方: local state と lifting up）
  • 概要: 状態は必要最小限の親に置き、propsで受け渡す原則を徹底する。

  • ミニ課題: 2つの兄弟コンポーネントの重複状態を親へ集約して挙動差分を確認する。
  • 公式リンク: Sharing State Between Components。

  • 概要: 判別可能Unionと型ガードで、状態分岐を安全に書く練習をする。

  • ミニ課題: 非同期状態オブジェクトをUnion化し、switchのdefaultにneverチェックを追加する。

Python実用活用（CLIツール化で再利用）
  • 概要: 一度書いた処理をCLI化して、毎日のデータ整形タスクを使い回せるようにする。
  • なぜ面白いか: 小さな自動化が積み上がると、手作業依存が一気に減る。
  • 現場での使いどころ: ログ整形、CSV集計、APIレスポンスの検査を定型コマンド化。
  • ハマりどころ: 引数バリデーション不足で本番ファイルを誤上書きしやすい。
  • ミニ課題: 既存スクリプトをargparse対応にし、入力ファイル存在チェックを追加する。
  • 公式リンク: argparse / pathlib / logging。
import argparse
from pathlib import Path

p = argparse.ArgumentParser()
p.add_argument('--input', required=True)
args = p.parse_args()
assert Path(args.input).exists(), 'input not found'

Zapier + n8n比較（導入速度と制御性の現実的な線引き）
  • 概要: Zapierは導入速度、n8nは分岐制御と自己ホスト性で優位という前提で選定する。
  • なぜ面白いか: ツール選定が「好み」ではなく、運用要件で説明できるようになる。
  • 現場での使いどころ: MVPはZapier、本番運用の複雑連携はn8nへ段階移行する設計。
  • ハマりどころ: 先に複雑化しすぎて運用者が触れなくなり、属人化する。
  • ミニ課題: 同じ連携（フォーム→DB→通知）を両方で作り、初期工数と保守工数を比較する。
  • 公式リンク: Zapier Platform / n8n Docs。

参考: Zapier Apps / n8n Docs

OpenClaw（cron + sub-agentの役割分担）
  • 概要: 定時配信はcron、重い分析はsub-agentへ分離して、主セッションを軽く保つ。
  • なぜ面白いか: タスクの責務分離が進み、失敗時の切り分けと再実行が楽になる。
  • 現場での使いどころ: 日次レポート生成、ログ分析、通知配信をワークフロー化する時に有効。
  • ハマりどころ: 監視頻度や通知先が過多で、重要アラートが埋もれる。
  • ミニ課題: 1タスクを「cron起動→sub-agent処理→Discord通知」に分離して運用メモを残す。
  • 公式リンク: OpenClaw Docs / GitHub。

参考: OpenClaw Docs / OpenClaw GitHub
